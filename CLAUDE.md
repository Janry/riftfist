# Проєкт: Beat 'em up (робоча назва)

Контекст-передача з дизайн-сесії. Це постійні інструкції для Claude Code.
Тримати цей файл коротким; деталі — в `ROADMAP.md` і в прототипі.

## Що це
2D side-scrolling beat 'em up у стилі Double Dragon / Battletoads.
Сюжет — подорожі по вимірах (оммаж до «Рік і Морті»), кожен вимір = рівень.
Ціль платформи: **iPhone (iOS)**, керування — тач.

## Жорсткі обмеження (не порушувати без явного рішення)
- **IP / ліцензія:** герой рендериться з наданих власником піксель-картинок
  (`rick1.jpg`/`rick2.jpg` — idle/кліпання; `rick3.png` — аркуш ударів) — впізнаваний
  R&M-арт. HUD-ім'я лишаємо пародійним («RIQ»). IP-залежну косметику (імена/кольори/
  теми вимірів) тримаємо в data-шарі `THEME`, окремо від рушія.
  **Рішення (2026-05-31, явне, власник):** одна шкіра героя з реальних картинок —
  без альтернатив і без живого перемикача арту. Ризик по ліцензії (Adult Swim
  фан-гру не ліцензує) власник приймає свідомо.
- **Кооп vs тач:** локальний кооп на одному екрані айфона не лізе (нема місця
  під 2 набори контролів). Рішення відкладене: або соло на телефоні, або
  кооп на десктоп/онлайн. Не закладати локальний 2P на одному пристрої.

## Поточний статус
**Рішення: browser-first.** Перед портом у Godot докручуємо контент і дизайн
прямо в браузері — ітерація дешева й миттєва. Тому `beatemup-prototype.html`
більше НЕ одноразовий: це активна збірка + жива специфікація (відчуття + контент).
Поверх підтвердженої механіки вже зроблено: баланс винесено в data-шар; ростер
ворогів (архетипи + поле `behavior`, дальник зі снарядами); структура виміру
(intro → хвилі → міні-бос → бос → «зачищено» → перехід); виміри як дані
(`THEME.dimensions`). **Арт героя:** рендериться з реальних картинок (`HERO_SKINS.rick`
= `imgFrameSrc` idle/кліпання + `atkSrc` аркуш ударів на 7 поз: джеб/прямий, зілля-спец).
Рушій сам прибирає фон (заливка), масштабує по контенту, білий флеш на влучанні;
старт чекає завантаження картинок (без блимання).

## Найближче завдання
Наповнювати контент у браузері (browser-first), доки гра не доведена як ПРОДУКТ
на коротку сесію (≈ вертикальний зріз із `ROADMAP.md`, але поки без арт-пайплайну).
Далі — баланс зброї/прогресії та поліш відчуття; і лише ПОТІМ порт у **Godot 4
(GDScript)** зі збереженням game feel. Порядок порту (коли дійде): рух у глибину →
автомат станів гравця + хітбокси → hitstop/тряска → вороги (AI) → стрибок/повітряний
удар → зброя → knockdown → бос.

## Підтверджена механіка (зберегти відчуття при порті)
Псевдо-3D: x = горизонталь, y = глибина (рух по y повільніший ×0.7),
сортування спрайтів по y. Влучання вимагає збігу і по x, і по глибині.

Ключові «ручки» відчуття (з прототипу — звіряти точні числа там):
- **hitstop**: 4–10 кадрів заморозки логіки на влучання (КРИТИЧНО для соковитості).
- **screenshake** + білий флеш цілі на кожен удар.
- **комбо-ланцюг** a1→a2→a3 через вікна скасування; 3-й удар важкий.
- **спец** коштує HP; **стрибок**: GRAV=0.8, JUMP_V=14; у високому стрибку
  ворожі удари не влучають (z>22) — стрибок і оборонний.
- **зброя** (data-шар `WEAPONS`): підбір на дотик, озброєний удар = мах,
  озброєний спец = кидок-снаряд; міцність у «зарядах», ламається.
- **knockdown**: важкі атаки (a3, спец, повітряний, кидок; прапор `down`)
  валять ворога; слабкі — лишають на ногах. Бос НЕ валиться (важкий).
- **бос**: фінал кожного виміру (після N хвиль + міні-бос), більше HP, довша
  дистанція, своя смужка.

## Конвенції
- Зберігати поділ THEME (дані) / ENGINE (механіка) і в Godot.
- Числа балансу й контент — у data-шарі, не «вшиті» в код. У прототипі це:
  `THEME` (косметика/IP), `BALANCE` (ручки відчуття), `WEAPONS`, `ATTACKS`,
  `ENEMIES` (архетипи + `behavior`), `THEME.dimensions` (виміри), `HERO_SKINS`
  (герой як дані: посилання на картинки — кадри idle/кліпання + аркуш ударів).
  Новий ворог, вимір чи герой = рядок даних, не правка рушія.
- **git-повідомлення — англійською.** Працюємо на `main` (без фіча-гілок).
- Оновлювати цей файл і `ROADMAP.md`, коли змінюються рішення чи статус.
